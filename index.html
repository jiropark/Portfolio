현재 개발 중인 UE5 ARPG 프로젝트입니다. 기존 프로젝트에서 경험한 구조적 문제들을 해결하기 위해 Host / Composer / FSM Runtime / Module+Blackboard의 4계층 아키텍처를 직접 설계하여 적용하였습니다. GameplayMessageSubsystem 기반의 메시지 버스를 통해 모듈 간 직접 의존성을 제거하고, 각 계층의 책임 범위를 명확히 분리하여 유지보수성을 높였습니다.

▸ Messaging 기반 모듈 통신 구조
설계 원칙
  • P2 모듈 간 직접 include 금지 — 모든 통신은 GameplayMessageSubsystem 경유
  • P7 Messaging 구독(RegisterListener)은 반드시 Composer의 Wire()에서만
  • P12 Host(ACharacter)는 게임 로직 없음 — 물리/메시만 소유
  • P13 Host API 호출은 Composer만 가능 — 모듈에서 직접 Host 접근 금지
  • P14 각 모듈은 자신이 소유한 Blackboard 필드만 읽기/쓰기
구현 규모
  • PlayerComposer: 모듈 20개, Messaging 핸들 41개
  • MonsterComposer: 모듈 3개 (CombatTarget / Movement / Animation)
  • FSM Bundle+Gate 27개 등록 — 상태 전환 인터럽트 처리
  • UXAmirCharBlackboard (per-character) + UXAmirGlobalBlackboard (크로스-캐릭터)
  • IBundle / IGate 인터페이스 기반 FSM Runtime 추상화
